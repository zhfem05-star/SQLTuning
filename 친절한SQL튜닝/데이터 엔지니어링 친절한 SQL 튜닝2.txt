데이터 엔지니어링 친절한 SQL 튜닝2

2장 인덱스 기본

I 인덱스 구조 및 탐색

1. 인덱스 기초 지식
(0) OLTP 시스템에서는 소량 데이터를 주로 검색하므로 인덱스 튜닝이 무엇보다 중요
(1) 인덱스는 데이터베이스를 효율적으로 찾기 위한 목차(색인)으로 기본적인 테이블 데이터 외에 따로 생성되는 데이터
(2) 보통 B tree 형태로 구성
(3) 인덱스는 루트블록, 브랜치블록, 리프블록으로 구성되고 각 블록은 용량 제한이 있음
(4) 인덱스를 활용할 경우 데이터의 스캔 순서는 루트블록 -> 브랜치블록 -> 리프블록 -> 테이블블록 순으로 ROWID를 받아 테이블블록으로 가게 됨
(5) 인덱스에도 값은 있기에 반환하는 값만 인덱스의 컬럼과 동일하다면 리프블록에서 값을 갖고 반환도 가능(테이블블록까지 가지 않아도 됨)
(6) 인덱스는 기본적으로 정렬되며 2개 이상일 경우 순서대로 정렬이 됨
(7) 인덱스는 주로 수직적 탐색, 수평적 탐색 두 과정을 통해 스캔

2. 인덱스 튜닝의 두 가지 핵심요소
(1) 인덱스 스캔 과정에서의 최적화
 1) 찾고자 하는 레코드 집합에 있어서 최대한 반환하는 레코드 개수가 적고 유니크한 조건을 갖는 컬럼을 인덱스 조건으로 설정하는 것이 좋음
(2) 테이블 엑세스 횟수 최적화
 1) 인덱스 스캔 후 테이블 레코드를 엑세스할 때 랜덤 I/O 방식을 사용하기에 랜덤 엑세스 최소화 튜닝이 중요
 2) 랜덤 I/O는 단순하게 조건에 맞는 레코드를 하나 찾기 위해 그에 맞는 블록에 찾아가는 방식
 3) 만약 데이터가 물리적으로 따로따로 다른 블록에 흩어져 있다면 랜덤 I/O 작업이 무수히 많게 시행됨. 즉 데이터를 한 번에 처리하지 못하고 하나씩 매번 처리해야 함
 4) 따라서 인덱스에서 리프블록을 통해 Range Scan을 하여 최대한 랜덤 I/O작업을 줄이는 것
(3) 둘 중 더 중요한 부분이 있다면 테이블 엑세스 횟수 최적화가 제일 좋음. 필자는 SQL 튜닝이 랜덤 I/O와의 전쟁이라 표현

3. 인덱스 구조
(1) 데이터베이스에선 인덱스 없이 검색하려면 처음부터 끝까지 데이터를 읽어야 함
 1) 인덱스를 사용하여 범위 탐색(Range Scan)을 통해 일부만 읽고 멈추기 가능
(2) 인덱스는 루트, 브랜치, 리프 블록의 트리 형태로 구성
 1) 각 블록에는 기본적으로 레코드가 들어 있으며 레코드는 하위 블록에 대한 주소값을 갖고, 키값은 하위 블록에 저장된 레코드들의 범위를 갖음
 2) LMC(Leftmost Child)는 키값을 갖지 않는 특별한 레코드로 키값을 갖는 첫 번째 레코드를 기준으로 그 값보다 작은 값들의 범위를 갖는 하위 블록을 가르키며 보통 가장 왼쪽 리프 블록으로 표현됨
 3) 각 블록은 용량 제한이 있음. 따라서 중복된 데이터(정확히는 인덱스 컬럼 기준으로 중복된 값)가 있는 경우 리프 블록이 달라도 경계값을 기준으로 데이터가 겹쳐있는 것 처럼 보일 수 있음

4. 수직적 탐색
(1) 의미: 인덱스 스캔 시작지점을 찾는 과정
(2) 즉 각 블록에서 조건을 만족하는 첫 번째 레코드를 찾는 과정

5. 수평적 탐색
(1) 의미: 데이터를 찾는 과정
(2) 리프블록은 양방향 연결 리스트로 좌에서 우, 우에서 좌로 탐색 가능(후에 인덱스를 통한 정렬 최적화에서 나옴)
(3) 목적은 조건절을 만족하는 데이터를 모두 찾기와 ROWID 얻기 두 가지

6. 결합 인덱스 구조와 탐색
(1) 인덱스 컬럼은 두 개로 설정이 가능하며 보통 선행된 컬럼을 기준으로 먼저 정렬이 되고 그 다음 컬럼을 기준으로 정렬됨
(2) 일반적인 조건 탐색과 다른 점
 1) 프로그래밍에서 OR나 AND를 사용할 때 순서에 따라 성능이 갈리지만 인덱스에선 선두 컬럼의 '=' 조건에선 순서에 따른 성능 차이가 나지 않음
 2) 이유는 간단하게 인덱스의 블록은 데이터를 범주화하여 카테고리로 만들지 않음. 리프 블록 용량에 맞게 정렬하여 데이터를 넣고 이를 기준으로 부모 블록에서 첫 번째 레코드를 찾아 내려오는 것. 결국 두 값을 모두 확인하고 그 범위가 시작되는 시점과 끝나는 지점을 찾는 것은 동일하기 때문임
 3) 반대로 만약 선두 컬럼이 범위 조건(>,<, Between)일 경우 후행 조건 컬럼의 데이터가 모두 흩어져 있는 경우가 발생하여 인덱스를 범위 스캔이 아니라 전부 스캔해야하는 불상사가 생김

II 인덱스 기본 사용법

1. 인덱스를 사용한다는 것
(0) 인덱스 기본 사용법: 인덱스를 Range Scan하는 방법을 의미. 인덱스를 Range Scan 할 수 없게 되는 이유를 알고 나면 인덱스를 Range Scan하는 방법도 자연스럽게 터득 가능
(1) 인덱스는 기본적으로 선두컬럼(처음으로 지정하는 인덱스 컬럼)을 기준으로 정렬이 되어 있음
(2) 따라서 선두컬럼을 기준으로 찾는 것은 Range Scan이 가능함. 정렬되어 있기에 데이터가 모여 있기 때문
(3) 인덱스를 정상적으로 정상적으로 사용: 리프 블록에서 스캔 시작점을 찾아 중간에 멈추는 것(range scan)
(4) 인덱스 컬럼을 가공(strsub 같은 기능)해도 인덱스를 사용하긴 하지만 Full Scan을 시행하거나 옵티마이저 판단하에 Table Full Scan을 할 수도 있음

2. 인덱스를 Range scan 할 수 없는 이유
(1) 인덱스 컬럼을 가공하면 정상적으로 사용 불가능
(2) 인덱스 스캔의 시작점을 알 수 없기 때문
 1) 보통 substr같은 인덱스 컬럼의 가공, 혹은 선두컬럼이 아닌 인덱스를 사용하는 경우 이런 현상이 야기됨
 2) 하지만 의외로 시작 값만 가공하고 그걸로 scan하는 경우에는 range scan이 가능(ex. LIKE '대한%'). 결국 시작점을 찾을 수 있는 경우이기에 가능한 것
 3) OR도 마찬가지며 서로 다른 컬럼을 쓰는 경우 어느 한 시작점을 찾을 수가 없기에 Range Scan 불가능
  a. 하지만 예외적으로 union all을 사용하여 쿼리문을 두 개로 나누면 인덱스 컬럼을 Range Scan 할 수 있음. 이는 Range Scan을 서로 다른 인덱스를 사용하는 방법으로 이를 OR Expansion이라 함
 4) IN 또한 Range Scan이 불가능하며 사실상 OR의 또 다른 형태이기에 불가능. 하지만 동일하게 OR Expansion을 써서 Range Scan 가능
  a. 그래서 IN 조건절에는 IN-List Iterator 방식을 사용. IN-List의 갯수만큼 Range Scan을 반복. 이를 통한 Union all과 동일한 성능을 냄

3. 인덱스 사용 조건
(1) 인덱스는 선두컬럼을 기준으로 먼저 정렬되기에 나머지 인덱스 컬럼들을 기준으로 스캔하게 되면 결국에는 Full Scan이 시행 될 수 밖에 없음
(2) 선두컬럼으로 Range Scan이 잘 되었다 하더라도 그 다음 인덱스도 범위 스캔하지 좋지 않다면 결국 효율적인 Range Scan이라 할 수 없음. 예를 들면 선두 컬럼은 제대로 되어 있지만 후행 조건이 LIKE '%PING%'이거나 substr을 쓴 경우 결국 선두컬럼 범위 내의 데이터를 Full Scan해야 하기 때문

4. 인덱스를 이용한 소트 연산 생략
(1) 인덱스는 선두컬럼을 기준으로 정렬하기에 order by가 없음에도 정렬된 결과가 나오게 됨(이를 옵티마이저가 반영)
(2) 내림차순에도 똑같이 사용가능. 이는 리프블록이 양방향 연결 구조이기에 가능. 오름차일 때는 우측으로 수평적 탐색, 내림차일 때는 왼쪽으로 수평적 탐색을 시행

5. ORDER BY 절에서 컬럼 가공
(1) 조건절 아닌 ORDER BY 또는 SELECT-LIST에서 컬럼을 가공해도 Range Scan이 안 되는 경우가 존재
 1) Concat하여 정렬을 하는 경우: index가 a와 b라는 컬럼 순으로 정렬되어 있으면 ORDER BY a, b를 할 시 정렬연산이 생략되지만 ||을 써서 두 컬럼을 붙이는 가공을 하게 될 경우 계산을 다시 해야 함
 2) 만약 SELECT 문에서 as를 잘못 주는 경우 생기는 휴먼 에러 또한 존재. 인덱스의 선두컬럼을 기준으로 order by를 준 것이 아닌 가공 컬럼을 기준으로 order by를 실행하면 아무리 인덱스를 잘 주었다고 생각해도 order by 연산이 생길 수 밖에 없음

6. SELECT-LIST에서 컬럼 가공
(1) 인덱스 a, b, c 순으로 주고 a하고 b인덱스를 사용가능하게 주어졌을 경우 SELECT 문에서 MIN같은 통계 연산 함수를 사용해도 SORT를 시행하지 않음
(2) 하지만 만약 SELECT에서 TO_NUMBER 등을 사용하여 데이터형 변환을 시행할 경우 기존의 정렬인 문자열 정렬에서 변환이 일어나게 될 경우 SORT 연산이 일어날 수 밖에 없음
 1) 정확히는 문자열 정렬 순서와 숫자 정렬 순서가 달라지는 경우에만 SORT 발생. 예를 들어 '1', '2', '10'을 문자열로 정렬하면 1, 10, 2 순이지만 숫자로 정렬하면 1, 2, 10 순이라 결과가 달라짐

7. 자동 형변환
(1) 인덱스에 해당하는 컬럼의 데이터를 바꾸는 경우, 특히 문자열과 숫자 형태의 형변환의 경우 서로 정렬 기준이 다르기에 Range Scan이 불가능
(2) 만약 특정 컬럼이 문자열인데 이를 생각하지 않고 where 조건문으로 숫자형 필터를 준 경우 옵티마이저는 Table Full Scan을 시행하게 됨. 인덱스 컬럼을 가공했기에 인덱스를 사용 못하고 모든 레코드들과 비교하는 상황까지 감
(3) DB는 날짜형 데이터의 경우 필터를 문자 그대로 넣어도 Full Scan이 아닌 Range Scan을 해주는 경우가 있는데 이는 DB가 날짜형 데이터를 우선하기 때문. 하지만 데이터 형태는 인덱스 컬럼을 가공 여부가 달린 매우 중요한 일이기에 꼭 성능에 문제가 없더라도 형 변환은 엄격하게 할 필요가 있음
(4) 형 변환 기능을 매번 사용해서 쓸데없는 연산이 된다고 착각할 수 있지만 어짜피 옵티마이저가 보고 형 변환을 시행하기에 생략한다고 해서 연산횟수가 주는 것도 아님. 따라서 꼭 WHERE의 조건에 사용할 때 명시적으로 컬럼의 데이터 형태와 비교하는 값의 데이터 형태는 일치 시켜줘야 함
