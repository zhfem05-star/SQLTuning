데이터 엔지니어링 친절한 SQL 튜닝1

1장 SQL 처리 과정과 I/O

I SQL 파싱과 최적화

1. 구조적, 집합적, 선언적 언어 
(1) SQL은 구조적 질의 언어로 정의에 따르면 구조적이고 집합적이고 선언적인 언어임
(2) DB에서 원하는 집합의 결과를 출력하기 위해 선언을 하여 데이터를 꺼내지만 만드는 과정은 절차적일 수 밖에 없음
(3) 하지만 선언적 언어이기에 일반 프로그래밍 처럼 컴퓨터에 시스템에 대한 세부적인 설정을 하는 것이 아니라 구조적 논리에 따라 데이터를 불러옴
(4) 그렇기에 데이터를 불러오는 코드를 짜주는 것은 옵티마이저가 대신 하며 SQL 최적화는 이런 옵티마이저가 DBMS 내부에서 프로시저를 작성하고 컴파일하여 실행 가능하게 만드는 전 과정을 의미
 1) DB를 예전에는 SQL 없이 직접 프로그래밍 언어를 통해 조회했지만, 매번 바뀌는 레코드와 테이블, 업데이트를 해야하는 데이터들이나 도메인 설정 등등 프로그래머들이 이런 가변적인 환경에서 일일히 코딩하여 조회하는 것은 매우 비효율적. 이에 따라 옵티마이저라는 하나의 로직을 통해 데이터를 조회하는 방식을 만들게 됨

2. SQL 최적화
(1) SQL 최적화 과정
 1) 클라이언트으로 부터 SQL 수신 -> 파싱 트리 생성 -> Syntax/Semantic 체크 -> SQL 최적화 -> 로우 소스 생성
 2) SQL 최적화: 미리 수집한 통계정보에 기반하여 다양한 실행 경로들 중 효율적인 하나를 택하여 실행
 3) 로우 소스 생성: 옵티마이저가 선택한 실행경로를 실제 실행 가능한 코드 또는 프로시저 형태로 변환하는 단계. 로우 소스 생성기가 그 역할을 맡음

3. SQL 옵티마이저
(1) SQL 옵티마이저는 사용자가 원하는 작업을 가장 효율적으로 수행할 수 있는 최적의 데이터 엑세스 경로를 선택해 주는 DBMS의 핵심 엔진
(2) 동작 방식
 1) 사용자로부터 전달받은 쿼리를 수행하는데 후보군이 될만한 실행계획들을 탐색
 2) 데이터 딕셔너리에 미리 수집해 둔 오브젝트를 통계 및 시스템 통계정보를 이용해 각 실행계획의 예상비용을 산정
 3) 최저 비용의 계획 선택

4. 실행 계획과 비용
(1) SQL 옵티마이저는 네비게이션과 비슷
(2) 실행 경로를 미리 볼 수 있는데 Execution Plan을 통해 볼 수 있음
(3) 미리보기 기능을 통해 인덱스를 스캔하는지 아닌지, 인덱스를 스캔한다면 어떤 것을 스캔하는지 확인 가능
 1) 여기서 계획의 기준은 비용이며 비용은 쿼리를 수행하는 동안 발생할 것으로 예상되는 I/O 횟수 또는 예상소요 시간을 표현한 값
 2) 하지만 만능은 아님 그래서 인덱스 힌트를 주는 경우도 있음

II SQL 공유 및 재사용

1. 소프트 파싱 VS 하드 파싱
(1) SQL은 데이터를 찾기 전 이전에 실행한 기록이 있으면 그걸 캐시로 저장하고 거기서 먼저 찾게 됨(찾은 데이터를 저장하는 게 아니라 실행한 SQL을 저장하는 것. 라이브러리 캐시라 불리는 공간에 저장함)
(2) 위와 같은 과정으로 찾으면 소프트 파싱. 그게 아니면 처음부터 파싱 전 과정을 다시 수행하면 하드 파싱
(3) 하드 파싱은 무거운 작업으로 이러한 라이브러리 캐시가 필요한 이유가 되기도 함
 1) 하드 파싱에 옵티마이저가 사용하는 정보
  a. 테이블, 컬럼, 인덱스 구조에 관한 정보
  b. 오브젝트 통계: 테이블 통계, 인덱스 통계, 컬럼 통계
  c. 시스템 통계; CPU 속도, Single Block I/O 속도, Multiblock I/O 속도
  d. 옵티마이저 관련 파라미터
(4) 라이브러리 캐시: SQL을 시행할 때 반복적으로 자주 실행되는 SQL의 경우 여기에 저장되어 빠르게 실행되게 가능하게 해줌
(5) 버퍼 캐시: 찾은 데이터를 미리 캐시 저장소에 넣어 다음에 데이터를 또 찾을 때 먼저 확인하는 곳

2. 바인드 변수의 중요성
(1) 사용자 정의 함수/프로시저, 트리거, 패키기지 등은 생성할 때 부터 이름을 갖지만 SQL은 이름이 따로 없음
(2) 전체 SQL 텍스트가 이름 역할을 함 -> 이는 SQL 쿼리문을 해싱을 통해 따로 구분하여 그럼
 1) 모든 쿼리는 해시 함수를 거쳐서 SQL ID를 생성
 2) 조금만 값이 달라져도 해시값이 변하기에 같은 내용이더라도 다르게 저장됨
(3) 예를 들어 유저가 로그인을 하는 상황을 가정
"""
"SELECT * FROM CUSTOMER WHERE LOGIN_ID ='"+ login_id +"'"
"""
쿼리문을 실행할 때 파라미터마다 다른 값을 받는 기능을 구현하면 DB에선 파라미터 별로 쿼리문을 생성하여 하드 파싱을 진행함 -> 성능 저하 발생
(4) 따라서 바인드 변수로 쿼리문을 실행하고 바인드 변수 안에 유저들의 로그인 ID를 받는 형태로 진행하면 하나의 쿼리문만을 라이브러리 캐시에서 재사용하는 형태로 최초 1회만 하드 파싱이 진행되고 이후에는 소프트 파싱으로만 진행

III 데이터 저장 구조 및 I/O 메커니즘

1. SQL이 느린 이유
(1) SQL이 느린 이유는 십중팔구 I/O 때문. 구체적으론 디스크 I/O 때문
(2) 필자는 이런 디스크 I/O를 프로세스가 잠에 드는 것이라고 설명
 1) 이러한 비유를 한 이유는 프로세스가 실행되는 동안 I/O가 필요한 작업 단계가 실행되면 프로세스는 잠깐 대기 상태로 전환하기 때문
 2) 프로세스의 생명 주기는 new -> (admited) -> ready -> (dispatched) -> running -> (exit) -> terminated의 부분에서 interrupted라는 부분을 만나게 되면 ready로 돌아가고, 만약 I/O 요청 또는 이벤트 대기가 존재할 경우 waiting 상태에 들어가게 됨
 3) 즉 I/O 작업에서 오래 걸리게 되면 프로세스 자체도 오래 걸릴 수 밖에 없는 것
 4) DB에서 SQL 최적화는 I/O 횟수를 최소화하거나, 버퍼 캐시를 활용해 디스크 I/O 자체를 덜 발생시키는 방법을 고민하는 것
(3) 필자는 디스크 I/O가 SQL 성능을 좌우한다고 해도 과언이 아니라 역설

2. 데이터베이스의 저장 구조
(1) 테이블스페이스: 세그먼트를 담는 콘테이너, 여러 개의 데이터파일(디스크 상의 물리적인 OS 파일)로 구성되며 데이터를 저장하려면 먼저 생성되어야 함
(2) 세그먼트: 인덱스처럼 데이터 저장공간이 필요한 오브젝트. 테이블, 인덱스를 생성할 때 데이터를 어떤 테이블스페이스에 저장할지를 지정. 여러 익스텐트로 구성됨
 1) 데이터파일: 테이블스페이스를 구성하는 물리적인 OS 파일. 실제 데이터가 디스크에 저장되는 단위. 하나의 테이블스페이스에 여러 개가 존재할 수 있지만, 하나의 데이터파일은 하나의 테이블스페이스에만 속함. 여러 세그먼트들을 담을 수 있음
 2) 테이블스페이스는 논리적 저장 관리 단위로, 테이블/인덱스를 생성할 때 저장 위치를 지정하기 위해 사용됨. 데이터파일은 물리적 개념으로 실제로 저장되는 파일을 지칭함. (ex. "CREATE TABLE 테이블명 TABLESPACE 테이블스페이스명")
 3) 세그먼트에 할당된 모든 익스텐트가 같은 데이터파일에 위치하지 않을 수 있음
 4) 즉 논리적으로는 같은 세그먼트에 속하지만, 물리적으로는 서로 다른 데이터파일에 흩어져 저장될 수 있음
(3) 익스텐트: 공간을 확장하는 단위. 연속된 블록들의 집합. 한 익스텐드에 담긴 블록은 모두 같은 테이블 블록
(4) 블록: 데이터를 읽는 단위. 아무리 작은 데이터를 읽고 싶어도 기본 블록단위만큼의 데이터가 스캔됨
(5) 하나의 테이블을 전체 조회 -> 익스텐드 내의 블록들을 모두 스캔
(6) 실습은 DBA_extent view 기능을 통해 실습

3. 블록단위 I/O
(1) 데이터는 기본적으로 아무리 작은 데이터라도 블록 단위로 읽게 됨
(2) 이는 테이블뿐만 아니라 인덱스도 블록 단위로 데이터를 읽고 씀

4. 시퀀셜 엑세스 vs 랜덤 엑세스
(1) 테이블 또는 인덱스 블록을 엑세스하는 방식으로는 시퀀셜 엑세스와 랜덤 엑세스, 두 가지가 있음
(2) 시퀀셜 엑세스: 논리적, 또는 물리적으로 연결된 순서에 따라 차례대로 블록을 읽는 방식
 1) 인덱스 리프 블록은 앞뒤를 가르키는 주소값을 통해 논리적으로 서로 연결되어 있음
 2) 이 주소 값에 따라 앞 또는 뒤로 순차적으로 스캔하는 방식이 시퀀셜 엑세스
 3) 오라클은 세그먼트에 할당된 익스텐트 목록을 세그먼트 헤더에 맵(map)으로 관리. 익스텐트 맵은 각 익스텐트의 첫 번째 주소 값을 갖게 됨
 4) 읽어야 할 익스텐트 목록을 익스텐트 맵에서 얻고 각 익스텐트의 첫 번째 블록 뒤에 연속해서 저장된 블록을 순서대로 읽으면 Full Table Scan
(3) 랜덤 엑세스: 논리적 물리적 순서를 따르지 않고, 레코드 하나를 읽기 위해 한 블록씩 접근하는 방식

5. 논리적 I/O vs 물리적 I/O
(1) DB 버퍼캐시
 1) 자주 읽는 블록을 매번 디스크에서 읽는 것은 매우 비효율적
 2) 라이브러리 캐시가 SQL과 실행계획, DB 저장형 함수/프로시저 등을 캐싱하는 코드 캐시라고 하면 DB버퍼캐시는 데이터를 캐싱하는 데이터 캐시
 3) 데이터 블록을 읽을 때는 항상 버퍼캐시부터 탐색
 4) 버퍼캐시는 공유 메모리 영역으로 같은 블록을 읽는 다른 프로세스도 득을 봄
(2) 논리적 블록 I/O는 SQL을 처리하는 과정에서 발생한 총 블록 I/O. 즉 읽어야 할 총 블록의 견적
(3) 물리적 블록 I/O는 디스크에서 발생한 총 블록 I/O(메모리에서 읽은 거는 아님). 즉 실제로 디스크에서 읽어야 하는 블록
(4) 메모리 I/O는 전기적 신호인데 반해 디스크 I/O는 엑세스 암(Arm)을 통해 물리적으로 일어나므로 메모리 I/O에 비해 상당히 느림
(5) 버퍼캐시 히트율(BCHR)
 1) 버퍼캐시 효율을 측정하는데 가장 전통적으로 많이 사용해 온 지표로 읽은 전체 블록 중 디스크 I/O를 수반하지 않고 곧바로 메모리에서 찾은 비율을 나타냄
 2) ((논리적I/O - 물리적I/O) / 논리적I/O) * 100 = 버퍼캐시 히트율
 3) 논리적 I/O는 일정하으로 물리적 I/O는 BCHR에 의해 정해짐
 4) BCHR은 시스템 상황에 따라 달라지므로 물리적 I/O는 외생변수, 즉 통제 불가능한 변수임
 5) 그렇기에 SQL 성능을 높이기 위해서 할 수 있는 일은 논리적 I/O를 줄이는 일
 6) 이를 위해 SQL 튜닝을 통해 읽는 총 블록의 개수를 줄여야 함
 7) 따라서 본 책은 SQL을 튜닝해서 논리적 I/O를 줄여 물리적 I/O도 줄여 성능을 향상하는 법에 대해서 다룸
(6) 실제 BCHR에 대한 정보를 조회하면 논리적 I/O를 구할 수 있음
 1) 디스크 I/O도 결국 버퍼캐시를 거쳐서 읽히기 때문에 Query에 포함됨
  a. Query가 버퍼캐시에서 읽은 데이터를 보여주는데 왜 Disk에서 읽어온 값들도 Query 안에 있냐. 이는 정확하게 말하면 Disk에서 읽어온 값도 결국 버퍼캐시에 적재하고 읽기 때문임

6. Single Block I/O vs Multiblock I/O
(1) Single Block I/O: 한 번에 한 블록씩 요청해서 메모리를 적재하는 방식
 1) 인덱스를 이용할 때는 기본적으로 인덱스와 테이블 블록 모두 Single Block I/O 방식을 사용
 2) 아래의 경우가 I/O 대상 오퍼레이션
  a. 인덱스 루트 블록을 읽을 때
  b. 인덱스 루트 블록에서 얻은 주소 정보로 브랜치 블록을 읽을 때
  c. 인덱스 브랜치 블록에서 얻은 주소 정보로 리브 블록을 읽을 때
  d. 인덱스 리프 블록에서 얻은 주소 정보로 테이블 블록을 읽을 때
(2) Multiblock I/O: 한 번에 여러 블록씩 요청해서 메모리에 적재하는 방식
 1) 많은 데이터를 읽을 때, 즉 테이블 전체를 스캔할 때 이 방식을 사용
 2) 대용량 테이블을 풀 스캔할 때 Multiblock I/O를 크게 설정하면 성능이 좋아짐
 3) 만약 읽고자 하는 블록을 DB 버퍼캐시에서 찾지 못할 경우 해당 블록을 디스크에서 읽기 위해 I/O Call을 함. 여기서 프로세스는 대기 큐에서 잠. 대용량 테이블일 경우 수많은 블록을 디스크에서 읽는 동안 여러 차례 잠을 잘 텐데, 기왕에 잠을 자려면 한 번에 많은 양을 요청해야 잠자는 횟수를 줄이고 성능을 높일 수 있음
 4) 특정 블록을 버퍼캐시에서 찾지 못하면 Multiblock I/O는 그 디스크 상에 있는 그 블록과 인접한 블록들을 한꺼번에 읽어 캐시에 미리 적재하는 기능
  a. 부연 설명을 하자면 인접한 블록은 같은 익스텐트에 속한 블록을 의미
  b. 즉 크기를 키워도 익스텐트의 경계를 넘지 못함
  c. 따라서 I/O Call 할 때의 블록 단위가 한 익스텐트와 딱 맞아 떨어지지 않아도 다른 익스텐트를 추가로 읽지 않고 부족한 상태로 읽어들이고 끝남
 5) DBMS의 블록사이즈가 얼마건 간에 OS 단에서는 보통 1MB 단위로 I/O를 수행
 6) 오라클에서는 이 단위를 정할 수 있음

7. Table Full Scan vs Index Range Scan
(1) Table Full Scan: 말 그대로 테이블을 전부 스캔하는 방식으로 테이블에 속한 블록 전체를 읽어서 사용자가 원하는 데이터를 찾는 방식
(2) Index Range Scan: 보통 아는 인덱스를 통한 탐색. 인덱스에서 일정량을 스캔하면서 얻은 ROWID로 테이블 레코드를 찾아가는 방식
 1) ROWID: 테이블 레코드가 디스크 상에 어디에 저장되었는지를 가르키는 위치 정보
  a. Row, 즉 레코드는 ROWID에 해당 데이터가 몇 번 파일, 몇 번 블록(이런 파일과 블록 번호에 대한 데이터를 담고 있는 것을 DBA라 함), 블록 안에서 몇 번 슬롯에 있는지를 담고 있음
  b. 즉 인덱스를 생성하는 과정에서 기준이 되는 값에 의해 모든 레코드들의 값들이 정렬되고 이걸 탐색하며 조건에 부합하는 레코드들의 ROWID를 통해 데이터를 탐색
(3) 인덱스를 이용하는데 성능이 느린 경우
 1) Table Full Scan은 Multiblock I/O를 통해 디스크 블록을 읽음. 블록에 속한 모든 레코드를 한 번에 읽어 들이고, 캐시에서 못 찾으면 I/O Call 한 번으로 인접한 수십 ~ 수백 개의 블록을 한꺼번에 I/O를 하는 메커니즘
 2) 시퀀셜 엑세스와 Multiblock I/O가 좋아도 소량의 데이터를 탐색할 때 매번 수십 ~ 수백의 데이터를 찾는 것 보단 인덱스를 통해 탐색하는 것이 훨씬 좋음
 3) Index Range Scan은 랜덤 엑세스와 Single Block I/O 방식으로 디스크를 읽음. 캐시에서 블록을 찾지 못하면 레코드 하나를 읽기 위해 매번 I/O Call을 해야하는 매커니즘. 따라서 많은 데이터를 읽어들일 때는 Table Full Scan보다 불리. 또한 읽었던 블록을 반복해서 읽는 문제점 또한 존재
 4) 따라서 읽을 데이터가 일정량을 넘으면 인덱스보다 Table Full Scan이 유리

8. 캐시 탐색 메커니즘
(1) Direct Path I/O를 제외한 모든 블록 I/O는 메모리 버퍼캐시를 경유. 대표적으로 아래의 케이스
 1) 인덱스 루트 블록을 읽을 때
 2) 인덱스 루트 블록에서 얻은 주소 정보로 브랜치 블록을 읽을 때
 3) 인덱스 브랜치 블록에서 얻은 주소 정보로 리프 블록을 읽을 때
 4) 인덱스 리프 블록에서 얻은 주소 정보로 테이블 블록을 읽을 때
 5) 테이블 블록을 Full Scan 할 때
(2) 버퍼 캐시의 구조
 1) 버퍼 캐시는 해시 함수를 통한 해시 체인(파라미터로 DBA를 받음), 그리고 데이터 블록으로 되어 있으며 해당 데이터 블록은 DBA(Data Block Address)가 있음
  a. 대충 이해를 위한 구조는 {해시체인:[(DBA, 슬롯번호)} 형태의 데이터라고 이해하면 됨
  b. 슬롯번호의 경우 버퍼 캐시에 들여온 데이터 블록을 읽을 때 Row에 접근하기 위해 필요. 즉 디스크에서 데이터를 읽어들일 때는 블록만 가져오기에 쓰지 않음
 2) 만약 인덱스를 통해 데이터를 탐색하게 될 경우 인덱스에서 데이터를 찾은 다음 해당 데이터 내에 있는 주소를 토대로 테이블 블록을 스캔. 먼저 버퍼 캐시에서 있는지 DBA를 통해 해시 함수를 통한 해시 체인 탐색 -> 그 안에 있는 블록을 탐색하고 없으면 이제 디스크에서 DBA 값을 통해 I/O Call하여 버퍼캐시에 올리고 읽고자 하는 Row의 슬롯 번호를 읽어들임
(3) 메모리 공유자원에 대한 직렬화
 1) 만약 버퍼 캐시를 전부 읽는 작업을 하는 와중에 다른 프로세스가 공유자원인 버퍼캐시에 접근하는 경우, 그리고 해당 버퍼 캐시의 체인 구조를 바꾸는 작업을 하게 되면 정합성 문제가 생김
 2) 따라서 순서에 맞게 메모리에 접근하도록 직렬화, 즉 줄 세우기를 해야 하며 이를 가능케 해주는 메커니즘이 래치
  a. 체인 래치: Transaction의 Lock의 개념과 유사(단 순서를 보장하지는 않음. 선착순이라 먼저 도착한 놈이 우선). 특정 프로세스가 버퍼 캐시에 접근할 경우 사용하려는 해시 체인에 Lock이 걸리고 자물쇠를 열 수 있는 키를 얻은 프로세스만이 진입 가능. 만약 해당 체인에서 블록을 찾았다면 래치를 해제 해야 함
 3) 체인 래치가 해시 체인에 대한 직렬화라면 버퍼 블록에 대한 직렬화도 존재. 이를 버퍼 Lock이라 함
  a. 버퍼 Lock: 만약 체인 래치를 풀었는데 후행 프로세스가 래치를 걸었던 프로세스와 같은 체인에 같은 블록에 작업을 하게 된다면 정합성 문제가 발생. 따라서 버퍼 헤더에 Lock을 걸어 버퍼 블록에 대한 직렬화 문제를 해결
 4) 따라서 직렬화를 통한 순서는 다음과 같음
  a. 프로세스의 데이터 접근 -> 체인 래치 획득 -> 해시 체인에서 블록 탐색 -> 블록 찾음 -> 버퍼 헤더에 블록 Lock -> 체인 래치 해제 -> 블록 읽기/수정 -> 버퍼 Lock 해제
