데이터 엔지니어링 친절한 SQL 튜닝3

3장 인덱스 튜닝

I 테이블 엑세스 최소화

1. 테이블 랜덤 엑세스
(0) 파티션 프루닝은 조건절에 해당하는 파티션만 읽는 기능이고 인덱스도 조건절에 해당하는 데이터를 읽는데 둘은 무슨 차이인가?
(1) 인덱스 ROWID는 물리적 주소? 논리적 주소?
 1) 우선 인덱스의 ROWID에는 어느 파일, 어느 블록, 블록 안에서 몇 번 슬롯에 있는지의 정보를 담고 있음
 2) 하지만 이는 DB 내에서 저장하는 정보 방식이지 메모리의 주소 개념인 포인터가 아님
 3) 정확히 말하자면 디스크 블록에 찾아가기 위한 간접적 주소이자 해시체인에 접근하기 위해 쓰이는 파라미터
(2) 메인메모리 DB와의 비교
 1) 메인 메모리 DB의 경우 말 그대로 DB의 데이터를 대부분 메모리에 저장하고 가져옴
 2) 지금까지 배운 이론상 버퍼캐시히트율 99%인 DB와 속도차이가 크게 나지 않아야 정상이지만 실제는 이와 다름
 3) 이유는 몇 가지가 있는데 아까 말한 ROWID 자체가 간접적인 주소이자 해시체인을 위한 파라미터이기 때문
 4) 기존 DB는 메모리에 있는 데이터는 수시로 밀려났다가 다시 캐싱되는 과정에서 동일한 포인터를 갖는 것이 불가능. 따라서 DBA를 해시체인의 간접적인 주소로 활용하여 메모리에 접근하고 해시체인 내에서 순차적으로 읽어들임
(3) I/O 메커니즘 복습
 1) 먼저 DB는 데이터가 버퍼에 있는지 확인을 함
  a. 해시 알고리즘의 파라미터로 DBA를 사용하여 해시 체인을 탐색
  b. 해시 체인 내에서 버퍼 헤더를 탐색
 2) 없으면 디스크에서 DBA를 통해 탐색. 이후 찾은 데이터를 버퍼에 올리고 읽어옴
 3) 이러한 과정 때문에 캐시버퍼 히트율이 높더라도 해싱과 래치 획득 과정을 반복해야 하기에 메인 메모리 DB보다는 느릴 수 밖에 없음
(4) 인덱스 ROWID는 우편주소
 1) 이 책의 필자는 ROWID는 우편 주소, 메인 메모리 DB의 포인터는 전화라고 비유함
 2) 물리적으로 연결된 전화는 빠르지만 우편 주소는 일일이 찾아다니면서 전해주는 구조이기에 느릴 수 밖에 없다는 비유

2. 인덱스 클러스터링 팩터
(1) 클러스터링 팩터(이하 CF): 특정 컬럼을 기준으로 같은 값을 갖는 데이터가 서로 모여있는 정도를 의미. 이는 물리적인 의미 또한 있음
(2) "거주지역 = 제주" 이런 데이터가 서로 물리적으로 몰려있으면 흩어져 있을 때 보다 데이터를 찾는 속도가 빠름(랜덤 I/O 비용이 준다는 의미. 이유는 후술)
(3) 인덱스 ROWID로 테이블 엑세스할 때 오라클은 래치 획득과 해시 체인 스캔과정을 거쳐 어렵게 찾아간 테이블 블록에 대한 포인터를 바로 해제하지 않고 일단 유지. 이를 버퍼 Pinning이라 함
(4) 이로 인해 다음 데이터를 찾을 때 같은 블록 안에 데이터가 있을 시 래치 획득, 해시 체인 과정이 생략되고 레코드를 바로 포인터로 접근
(5) 만약 CF가 좋지 않은 인덱스를 사용하면 매번 엑세스 할 때 마다 블록 I/O가 발생

3. 인덱스 손익 분기점
(1) 인덱스 ROWID를 이용한 테이블 엑세스는 생각보단 고비용 구조
(2) Index Range Scan에 의한 테이블 엑세스가 Table Full Scan보다 느려지는 지점을 인덱스 손익 분기점이라 부름
(3) 인덱스를 이용한 테이블 엑세스가 Table Full Scan 보다 더 느려지게 만드는 가장 핵심적인 두 가지 요인은 다음과 같음
 1) Table Full Scan은 시퀀셜 엑세스인 반면 인덱스는 랜덤 엑세스
 2) Table Full Scan은 Multiblock I/O, ROWID를 이용한 방식은 Single Block I/O
(4) 테이블이 클수록 손익분기점이 되는 비율이 낮아짐. 즉 대용량 테이블에서 인덱스가 유리한 구간은 전체 데이터 중 극히 일부를 추출하는 경우로 좁혀짐
(5) 위와 같은 개념은 OLTP와 OLAP 튜닝의 특징을 구분 짓는 핵심 개념이 됨
(6) 온라인 프로그램 튜닝 VS 배치 프로그램 튜닝
 1) 조회하는 데이터가 적은 경우 인덱스와 NL조인(이중 for문 구조의 조인)이 효과적
 2) 조회하는 데이터가 많은 경우 해시 조인과 Full Scan이 효과적
 3) 따라서 배치 프로그램에서는 파티션 활용 전략이 매우 중요한 튜닝 요소, 병렬 처리까지 더할 수 있으면 금상첨화
 4) 인덱스는 큰 테이블에서 아주 적은 일부 데이터를 빠르게 조회하고자 할 때 주로 사용

4. 인덱스 컬럼 추가
(1) 만약 탐색 조건이 2가지 이상이며, 선두컬럼 외의 조건을 인덱스에서 찾을 수 없는 경우 나머지 조건을 찾기 위해 인덱스에서 테이블 블록으로 랜덤 엑세스가 추가적으로 발생할 수 있음
(2) 하지만 인덱스에 조건에 대한 컬럼을 추가할 경우 인덱스 스캔 자체는 줄지 않지만 테이블에 일일이 들려 나머지 조건을 확인하는 랜덤 I/O 횟수는 줄일 수 있음
(3) 이는 인덱스가 색인 겸 데이터 조회의 역할 또한 가능함을 시사(커버링 인덱스)
(4) 클러스터링 팩터가 잘 된 인덱스의 경우 테이블 엑세스가 훨씬 덜 일어남. 같은 블록 내에 데이터가 있으며, 메모리에서 같은 블록을 버퍼캐시에서 재사용해서 테이블 엑세스, 즉 랜덤 I/O가 덜 발생

5. 인덱스만 읽고 처리
(1) 위에 설명한 커버링 인덱스의 개념이 사용될 수 있는 경우가 있음
(2) 만약 컬럼이 적은 데이터의 경우 인덱스에서 데이터를 미리 다 얻어서 성능을 올리는 방법을 사용 가능
(3) Include 인덱스
 1) 정확히 이 옵션은 인덱스와 같이 색인의 기준으로 넣는 것은 아니지만 마지막 인덱스 리프 레벨 노드에 추가적으로 넣을 데이터들을 고를 수 있음
 2) 원래는 인덱스로 지정된 데이터들만 인덱스에서 조회할 수 있지만 Include를 사용하면 인덱스 단위에서 인덱스 외의 데이터를 추가하여 확인 가능함
 3) 단 Include로 추가된 데이터들은 정렬을 보장 못하고 브랜치나 루트에서 색인 기준이 되지 않기에 순수히 테이블 랜덤 엑세스를 줄이기 위한 데이터 복사라고 봐야 함

6. 인덱스 구조 테이블
(1) 앞선 커버링 인덱스에서 영향을 받은 개념으로 그냥 모든 데이터를 싹다 인덱스 구조로 생성한 테이블을 인덱스 구조 테이블(이하 IOT)이라 함
(2) 일반 구조의 테이블을 힙 구조 테이블이라 하는데 DDL로 오라클에서 "organization index"라 하면 인덱스 구조로, "organization heap"이라 하면 기본 구조로 생성됨
(3) 힙 구조 테이블에서 데이터를 입력할 때는 랜덤 방식을 사용. 즉 블록에 정해진 순서 없이 데이터를 입력
(4) IOT는 인덱스 구조 테이블이라 정렬 상태를 유지하며 데이터를 입력(PK기준 정렬 상태를 유지하기에 PK가 비슷한 데이터의 물리적인 거리가 가까움)
(5) 따라서 이는 인위적으로 CF를 좋게 만드는 방법 중 하나

7. 클러스터 테이블
(1) 인덱스 클러스터 테이블은 클러스터 키 값이 같은 레코드를 한 블록에 저장하는 구조
(2) 한 블록에 다 담지 못하면 새로운 블록에 할당해서 클러스터 체인으로 연결
(3) 심지어 여러 테이블 레코드를 같은 블록에 저장 가능하며 이를 다중 테이블 클러스터라 함
(4) 하지만 같은 클러스터를 같은 블록에 저장할 뿐 정렬하지는 않음
(5) 클러스터에 테이블을 담기 전에 클러스터 인덱스를 반드시 정의해야 함. 클러스터 인덱스는 데이터 검색 용도로 사용할 뿐만 아니라 데이터가 저장될 위치를 찾을 때도 사용하기 때문(기존 인덱스는 저장을 빈 블록 아무데나 하고 나중에 그 위치를 ROWID로 기록하지만 클러스터 인덱스의 경우 애초에 저장할 때 클러스터 인덱스를 통해 위치를 정함)
(6) 클러스터 인덱스도 일반 B Tree를 인덱스 구조를 이용하지만 테이블 레코드를 일일이 가르키지 않고 키 값을 저장하는 첫 번째 데이터 블록을 가르킴. 만약 데이터가 해당 블록에 다 차있으면 클러스터 체인을 통해 다른 블록을 찾아감
(7) 일반 인덱스 레코드는 테이블 레코드와 1:1 관계이지만 클러스터 인덱스는 1:M의 관계를 갖게 됨
(8) 따라서 클러스터 인덱스의 키값은 항상 유니크함
(9) 해시 클러스터 테이블
 1) 인덱스를 사용하지 않고 해시 알고리즘을 사용해 클러스터를 찾아감
